# 設計メモ：KCL Progress Board（進捗タイムライン + Q&A AI + Webダッシュボード）

---

## 1. 何を解決したい？
- **困っていること（いま何が問題？）**
  - 参加者同士で進捗が見えず、切磋琢磨しづらい（「みんな今どこまで進んでる？」が分からない）
  - 質問したいのに「どこで/どう書けばいいか」分からず、質問が出ない or メンターに分散して埋もれる
  - Slack上で有益な議論や仕様決定が起きても、そのまま流れてしまいタスクとして残らない
- **だれが困る？**
  - 参加者：進捗の比較や刺激が得られない / 質問しづらく手が止まる / 決まったタスクを忘れがちになる
  - メンター：質問が分散・情報不足（ログや環境情報がない等）で対応コストが高い
  - 運営：各チームの全体状況が見えず、手当てが遅れる
- **うまくいったと言える条件（できれば数字）**
  - 進捗の可視化：参加者の80%以上が、1日2回（例：昼と夕方）の進捗共有を迷わずスムーズに行える
  - 質問導線：質問テンプレの利用率70%以上、初回返信（AI含む）までの中央値10分以内
  - 状況把握：運営がWebダッシュボードを見るだけで、全チームの状況を即座に把握できる
  - タスク化：Slackからの自動Issue起票が各チームで活用され、やるべきことの漏れが減る

---

## 2. スコープ定義（やらないこと / Nice to have）
- **今回やらないこと**
  - GitHub連携：初心者の学習コストが高く「ツールの使い方の学習」にすり替わるリスクがあるため今回は除外（※進捗の自動取得は行わず、SlackからのIssue起票アクションのみに留める）。
- **Nice to have（余裕があれば対応）**
  - 高度なデータ分析機能：データ自体は最初からRDB（および必要に応じたNoSQL）を用いて構造的に管理するが、それを活用した複雑なダッシュボード分析や永続的な履歴の追跡機能を作り込むことは後回しとする。

---

## 3. アイデア
- **こうする：**
  - **進捗管理**：参加者はSlackから `/progress` で指定のフォーマット（現在のフェーズ、SOS状態など）を入力し、Botが `#progress-board` に自動投稿。同時にWebダッシュボードにも連携して一覧化する。
  - **Q&A自動化**：Slackの `/question` コマンド等で開発者が質問すると、入力内容に対してAI（LLM）が「一次回答」と「不足情報の聞き返し（例：エラー文がありません等）」を行う。必要に応じてメンター用チャンネルへ転送する。
  - **Slack議論のIssue化**：Slack上で議論が起きた際、特定のアクション（絵文字やメニュー等）をトリガーに、内容を適切に要約してGitHubへ自動でIssue化する。
  - **Webダッシュボード**：Slackのタイムラインを「正」としつつ、運営・参加者が全チームの進捗を俯瞰できる専用Web画面を提供する（外部サイトへの過度な誘導はせず、あくまで閲覧用）。
- **なぜそうする？**
  1. 入力は使い慣れたSlack（フォーム）に集約し、指定フォーマット化することで学習コストと心理的ハードルを下げるため。
  2. AI一次回答とテンプレ補完を挟むことで、即時性を提供しつつメンターの対応コストを劇的に下げるため。
  3. Webダッシュボードを用意することで、時系列で流れてしまうSlackの弱点を補い、ハッカソン全体の「熱量」をひと目で実感できるようにするため。
  4. 初学者が手動でIssueを書く手間を省き、Slackの熱量ある議論をワンアクションで確実にタスク（ストック情報）に変換するため。

## 4. 導入によって生じるサークル内の変化
- **フローの変化：**
  - Before: 進捗は口頭や散発的な投稿で埋もれ、質問もDMや雑談に混ざって情報不足のまま飛んでくる。議論もSlack上で完結して忘れ去られる。
  - After: `/progress` で進捗がダッシュボードにまとまり、`/question` でAIが一次対応した整理済みの質問だけがメンターに届く。議論は適切にIssue化される。
- **サークル全体での変化：**
  - 運営・メンターは状況把握と一次対応に追われる時間が減り、より本質的な技術サポートや壁打ちに集中できる。
- **利用者個人の変化（心理的側面）：**
  - ダッシュボードで他チームの頑張りが見えて「自分たちも進めよう」という連帯感が生まれる。
  - 質問の入口が明確化＆AIが相手になることで、「こんなこと聞いていいのかな」という迷いがなくなる。

## 5. 作業の進め方
- **システム構成イメージ**
  - フロントエンド：WebダッシュボードUI（React等、シンプルな一覧画面）
  - バックエンド：Go等によるAPIサーバー。Slack Webhook（Event/Slash Command）の受け口と、DB操作、LLM API呼び出し、およびGitHub APIの実行を担当。インフラはCloud Runなどを想定。
  - DB：最初からRDB（PostgreSQL等）でチーム情報や進捗ログなどを適切に構造化して管理しつつ、AIの会話ログやSlackのペイロードなど柔軟性が求められる部分には適宜NoSQLを組み合わせる適材適所の設計とする。
- **開発ステップ（MVPアプローチ）**
  1. **ステップ1（進捗機能）**：Slack `/progress` コマンド → フォーマット入力 → Webhook受け取り → DB保存 → Webダッシュボード表示。
  2. **ステップ2（Q&A基盤）**：Slack `/question` コマンド → フォーマットを整えて指定チャンネルにBot投稿。LLM APIを用いた一次回答とヒアリングを実装。
  3. **ステップ3（AI連携・Issue化）**：Slackの特定アクション検知 → スレッド履歴取得 → LLMによる要約 → GitHub APIで自動Issue起票機能を実装。

## 6. 分からないこと・懸念事項
- **Slackワークスペースの権限**：Slackアプリのインストール、Slash Command作成、Event Subscriptionsの設定権限が運営アカウントにあるか。
- **LLMの選定とプロンプト調整**：一次回答の精度をどこまで求めるか。ハルシネーション（誤った回答）に対する免責事項の提示方法。
- **UI/UXのバランス**：WebダッシュボードとSlack、どちらをメインで見てもらうかの参加者へのアナウンス方法。
- **GitHub連携の仕様**：SlackからIssueを作成する際のリポジトリの紐付け方や、認証トークンをサーバーにどう持たせるか（ハッカソン用の簡易運用など）。
