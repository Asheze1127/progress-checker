# 📋 機能一覧テンプレート：KCL Progress Board

---

## 0️⃣ 前提定義（最初に決める）

| 項目      | 内容                                                                                                                                     |
| ------- | -------------------------------------------------------------------------------------------------------------------------------------- |
| 対象ユーザー  | ハッカソン参加者（主に初心者） / メンター / 運営                                                                                                              |
| 提供価値    | 参加者同士の進捗可視化による切磋琢磨の促進、質問ハードル低下とメンターの対応コスト削減、Slack上の有益な議論の確実なタスク化（ストック化）。                                                               |
| コア体験    | Slackのコマンド（`/progress`, `/question`）やアクションを通じたシームレスな入力体験、AIによる即時の一次回答、Webダッシュボードでの全体俯瞰。                                                |
| 成功指標    | ・進捗共有：参加者の80%以上が1日2回実施<br>・質問導線：質問テンプレ利用率70%以上、初回返信（AI含む）中央値10分以内<br>・運営状況把握：Webダッシュボードによる即時把握<br>・タスク化：Slackからの自動Issue起票の定着（漏れの削減） |

---

# 1️⃣ 機能カテゴリ設計（自由定義）

## 【カテゴリA：進捗管理と可視化（Progress Board）】

| #   | 機能名 | 概要 | ユーザー価値 | 優先度   | 備考 |
| --- | --- | -- | ------ | ----- | -- |
| A-1 | Slack `/progress` コマンド | Slackから指定フォーマット（フェーズ、SOS状態など）で進捗を送信する機能 | 参加者が使い慣れたツールで迷わず簡単に進捗を報告できる | 🔴 P0 | 入力ハードルを下げるための最重要動線 |
| A-2 | 進捗のSlack自動投稿 | 入力された進捗を各チームのSlackチャンネルにフォーマットを整えて投稿する機能 | 参加者同士で互いの進捗が見え、刺激や連帯感が生まれる | 🔴 P0 |  |
| A-3 | Webダッシュボード（一覧画面） | DBに保存された全チームの最新進捗を一覧表示するWeb画面 | 運営が全体状況を即座に俯瞰でき、遅れているチームへの手当てがしやすくなる | 🔴 P0 | シンプルな閲覧用UI（React等） |
| A-4 | チーム情報の構造化管理（DB） | チーム、ユーザー、進捗ログをRDB（PostgreSQL等）で構造的に保存する基盤 | データの整合性を保ち、将来的な機能拡張の土台となる | 🔴 P0 | 一部柔軟なデータはNoSQLを活用 |
| A-5 | 進捗タイムライン履歴表示 | 過去の進捗ログを時系列でさかのぼって確認できる機能 | チームの成長過程や、つまずいているポイントの期間を把握できる | 🟢 P2 | |

---

## 【カテゴリB：Q&Aサポート（AI連携）】

| #   | 機能名 | 概要 | ユーザー価値 | 優先度   | 備考 |
| --- | --- | -- | ------ | ----- | -- |
| B-1 | Slack `/question` コマンド | 質問用フォーマット（現状、試したこと等）を呼び出して入力する機能 | 「どう質問していいか分からない」という参加者の心理的ハードルをなくす | 🔴 P0 | テンプレ利用率70%以上の要 |
| B-2 | AI（LLM）による一次回答 | 入力された質問に対し、LLM APIを利用して即座に解決策のヒントを返信する機能 | 待ち時間をなくし、自己解決を促す（初回返信10分以内の達成） | 🟡 P1 | ハルシネーションの免責事項を添える |
| B-3 | AIによる不足情報のヒアリング | 質問にエラーログや環境情報が不足している場合、AIが聞き返す機能 | メンターが対応する際の情報不足を防ぎ、コミュニケーションコストを下げる | 🟡 P1 | |
| B-4 | メンターチャンネルへの転送 | 整理された質問内容とAIのやり取りログを、メンター専用チャンネルへ通知する機能 | メンターは状況を把握した上で、本質的なサポートのみに集中できる | 🔴 P0 | |

---

## 【カテゴリC：議論のタスク化（GitHub連携）】

| #   | 機能名 | 概要 | ユーザー価値 | 優先度   | 備考 |
| --- | --- | -- | ------ | ----- | -- |
| C-1 | Slackアクション検知（Issue化トリガー） | Slackメッセージの特定絵文字リアクションやメニュー操作を検知する機能 | ワンアクションでタスク化のフローを開始できる | 🟡 P1 | |
| C-2 | スレッド履歴の取得とAI要約 | トリガーされたメッセージとそのスレッドの会話履歴を取得し、LLMで要約する機能 | 議論の流れを人間が手作業でまとめる手間を省く | 🟢 P2 | |
| C-3 | GitHub 自動Issue起票 | 要約された内容をもとに、GitHub APIを叩いて指定リポジトリにIssueを作成する機能 | フロー情報（Slack）をストック情報（GitHub）に変換し、やるべきことの漏れを防ぐ | 🟢 P2 | 初学者の学習コストを回避 |
| C-4 | リポジトリ・認証情報の一元管理 | 各チームのSlackチャンネルとGitHubリポジトリ、認証トークンを紐付ける管理機能 | 運営側でのセットアップを容易にし、セキュリティを担保する | 🟢 P2 | |

# 3️⃣ フェーズ整理テンプレート

```text
【Phase 0 - MVP】
🔴 P0のみ実装
- Slackでの進捗入力（/progress）と一覧ダッシュボード表示
- Slackでの質問入力（/question）とメンターへの転送基盤
- 最低限のDB（RDB）設計と構築

【Phase 1 - 体験向上】
🟡 P1追加
- 質問に対するAI（LLM）の一次回答と不足情報のヒアリング
- Slack上の議論をタスク化するためのアクション検知基盤

【Phase 2 - 安定化】
🟢 P2追加
- SlackスレッドのAI要約とGitHubへの自動Issue起票機能
- チームごとの進捗タイムライン履歴表示
- GitHub連携の各種管理機能

【Phase 3 - 拡張】
⚪ P3追加
- 蓄積データを活用した高度なダッシュボード分析機能
```
