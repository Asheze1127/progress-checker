# 設計メモ：KCL Progress Board（進捗タイムライン + 質問導線 + AI一次回答）

---

## 1. 何を解決したい？
- 困っていること（いま何が問題？）：
  - 参加者同士で進捗が見えず、切磋琢磨しづらい（「みんな今どこまで進んでる？」が分からない）
  - 質問したいのに「どこで/どう書けばいいか」分からず、質問が出ない or 分散して埋もれる

- 既存サービスを使った解決策とその問題点
  - Slack：
    - 良い：全員いる
    - 悪い：進捗が時系列でまとまらない / 質問が流れて埋もれる / 書き方が分からない人がいる
  - GitHub（コミット/PR）：
    - 良い：進捗は事実として残る
    - 悪い：リポジトリが分散、見に行かないと分からない / 非エンジニアは追えない
  - Notion/Googleスプレッドシート：
    - 良い：一覧化できる
    - 悪い：更新が手動になりがちで続かない

- だれが困る？：
  - 参加者：進捗の比較や刺激が得られない / 質問しづらい
  - メンター：質問が分散・情報不足で対応が重い
  - 運営：全体状況が見えず、手当てが遅れる

- うまくいったと言える条件（できれば数字）：
  - 進捗の可視化：
    - 「誰が何を進めているか」を1画面で把握できる状態が、参加者の80%で達成
    - 進捗投稿（自動/手動含む）が 1チーム1日1件以上
  - 質問導線：
    - 質問の投稿先が迷わない（質問テンプレ利用率 70%以上）
    - 初回返信（AIまたはメンター）までの中央値：30分 → 10分
    - “情報不足”質問（環境/ログ/再現手順なし）率：60% → 25%

---

## 2. 今回やらないこと
- やらないこと：
  - 完璧なSNS（いいね/フォロー/DM/通知地獄）にはしない
  - ガチの権限管理・監査ログ・課金など運用SaaS化はしない
  - AIが最終回答を確定する仕組み（誤回答責任）まではやらない
  - GitHubの全イベント対応（Issue/Release/Actionsなど全部）はやらない（最初はPR/commit/README更新などに絞る）
  - モバイルアプリは作らない（Web + Slackで完結）

---

## 3. アイデア
- こうする：
  - 参加者向け「掲示板/タイムライン（Progress Board）」を用意し、チーム/個人の進捗が流れる
  - GitHub連携で、commit/PRなどを自動でタイムラインに流し込む（進捗の“自動生成”）
  - 質問は「質問専用導線」に集約（Slackの /question かフォーム）
  - GPTOSSを使って「一次回答 + 質問テンプレ補完（不足情報の質問返し）」を行い、必要ならメンターチャンネルへ転送

- なぜそうする？（理由は2〜3個まで）：
  1. 進捗入力を“努力”にすると続かないので、GitHubイベントで自動化して継続率を上げる
  2. 質問の場所と書き方を固定し、心理的/手続き的なハードルを下げる
  3. AI一次回答で即時性を作りつつ、最終判断は人（メンター）に繋げて品質を担保する

## 4. 他の案
- 案A：Slackだけで運用（進捗チャンネル + 質問チャンネル + ピン留めテンプレ）
  - 良いところ：最速・新規開発ほぼ不要
  - 悪いところ：進捗が時系列で埋もれる / GitHub進捗が見えない / 切磋琢磨が弱い
- 案B：GitHub Projects / Notionで進捗管理（全員が毎日更新）
  - 良いところ：一覧性が高い
  - 悪いところ：手動更新が前提で続かない / エンジニア以外が乗りづらい
- 今回選ばない理由（短く）：
  - 「継続運用」を考えると、手動中心やSlackのみは“見える化が弱い”か“続かない”になりやすい

## 5. 導入によって生じるサークル内の変化
- フローの変化：
  - Before:
    - 進捗：各チーム内で口頭/散発的、他チームは見えない
    - 質問：どこに書くか迷う → DM/口頭/雑談で終わる or 埋もれる
  - After:
    - 進捗：GitHub更新が自動でタイムライン化 + 追加で“今日やったこと”を一言投稿できる
    - 質問：/question or フォーム → AIが不足情報を聞き返し → 必要ならメンターへ転送

- サークル全体での変化(効率面/費用面)：
  - Before:
    - 運営/メンターが状況把握に時間を取られる（聞き回り）
    - 同じ質問が繰り返される
  - After:
    - 全体状況を1画面で把握でき、支援が早くなる
    - AI一次回答とテンプレで、メンター対応の“質”が上がる（情報不足が減る）
    - 追加コスト：ホスティング + LLM利用（小額に抑える設計）

- 利用者個人の変化(心理的側面)：
  - Before:
    - 「自分だけ遅れてる？」が分からず不安/孤立
    - 質問が怖い、書き方が分からない
  - After:
    - みんなの進捗が見えて刺激になる（切磋琢磨）
    - 質問の入口が明確で、まずAIに投げて心理的ハードルが下がる

## 6. 作業の進め方
> ポイント：先に“動く形”を作って、残りで直す（前倒しする）。

### 6.1 全体の見積もり（ざっくりでOK）
- メンバーごとに記載（例）
  - Frontend：タイムラインUI / 投稿UI / チームフィルタ
  - Backend：GitHub Webhook受け / DB / API
  - Bot/AI：Slackコマンド(/question) / GPTOSS呼び出し / テンプレ補完
  - Infra：デプロイ（Amplify/Vercel + API/Lambda等）/ 環境変数 / 権限設定

- 全体の作業時間：合計 18〜28時間（2〜3日想定）
- 使える日：平日2日 + 土日1日（想定）

### 6.2 最初にやること（1〜3日で終わるサイズ）
- まず作る最小版（MVP）：
  1. Webのタイムライン（DBに“進捗イベント”を入れて表示するだけ）
  2. GitHub Webhook（push だけ）→ イベントをDBに保存 → タイムラインに流れる
  3. Slackの /question → “テンプレを埋めさせる” → 1つの質問チャンネルに投げる
  4. GPTOSS一次回答（短く）+ 不足情報チェック（例：環境/エラー文/再現手順が無いなら質問返し）

- それで確かめたいこと：
  - GitHub連携が「入力コストゼロの進捗可視化」になっているか
  - 質問テンプレが“質問しやすさ”を上げるか（投稿数が増えるか）
  - AI一次回答が“即時性”と“情報補完”に効くか（メンター負担が減るか）

- できなかったらどうする（簡単な逃げ道）：
  - GitHub連携が間に合わない → Web上で「今日やったこと」手動投稿だけにする（後で連携）
  - AIが不安定 → AIは“不足情報チェック”だけにして、回答はメンターへ誘導
  - Webが間に合わない → Slackに「#progress」集約 + botでGitHubイベントを投稿（掲示板は後日）
