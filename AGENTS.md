# AGENTS.md

coding agent（Codex / Copilot Coding Agent / Cursor など）が作業するための汎用指示書。
**プロダクト固有の情報はここに書かない。** 仕様・設計・環境変数などは [`docs/details/`](./docs/details/) 配下を参照すること。

---

## 基本

- 日本語で応答すること
- 必要に応じてユーザに質問を行い、要求を明確にすること
- 作業後、作業内容とユーザが次に取れる行動を説明すること
- コマンドの出力が確認できない場合、get last command / check background terminal を使用して確認すること

---

## 開発フロー

作業を以下のフェーズに定義する。ユーザの指示キーワードに従ってフェーズを切り替える。

| 指示 | やること | 絶対にやらないこと |
|------|----------|------------------|
| 「調査」 | `docs/reports/YYYYMMDDHHmm-title.md` に調査結果を記載。不明点は fetch MCP で検索 | コードを書く |
| 「計画」 | `tasks.md` に計画を記載。コードベースと docs を読み、関連ファイルパスをすべて列挙。必要最小限の要件のみ記載。前回の内容が残っている場合は読まずに上書きしてよい | コードを書く |
| 「実装」 | `tasks.md` に記載された内容のみを実装する | 記載以上の実装・デバッグ |
| 「デバッグ」 | 直前のタスクのデバッグ「手順」のみ提示する | 勝手にコードを修正する |

## 返信・出力フォーマット

### ファイルへの言及（必ず守る）

ファイルやコードに言及するときは **必ずファイルパスを ref として添付する**。
「そのあたり」「該当箇所」などの曖昧な表現は禁止。

```
# 良い例
`backend/internal/handler/slack.go` の `HandleProgress` 関数を修正しました。

# 悪い例
ハンドラのファイルを修正しました。
```

コードブロックにはファイルパスを明示する：

````
```go:backend/internal/handler/slack.go
// コード
```
````

### 複数ファイルにまたがる変更

変更ファイル一覧を先頭にまとめて提示する：

```
## 変更ファイル
- `path/to/file_a.go` — 変更内容の一言説明
- `path/to/file_b.ts` — 変更内容の一言説明
```

### コードレビューの返し方

レビューコメントは以下の形式で書く：

```
[重要度] path/to/file.go#L行番号
問題の説明。（必要なら修正案）
```

重要度の定義：

| ラベル | 意味 | 対応 |
|--------|------|------|
| `[MUST]` | バグ・セキュリティ・規約違反 | マージ前に対応必須 |
| `[SHOULD]` | 可読性・保守性の改善 | 対応推奨 |
| `[NIT]` | 細かい好みの問題 | 対応任意 |
| `[QUESTION]` | 意図確認 | 回答を求める |

## コーディング規約

### 共通

- 関数・変数名は英語。コメントは日本語・英語どちらでも可
- マジックナンバーは定数化する
- TODO / FIXME は `TODO(名前): 内容` の形式で書く
- ファイルを新規作成する前に、既存ファイルで対応できないか確認する
- 不要なコメント（「ここでXXをする」などコードをそのまま説明するだけのもの）は書かない

### Go

- エラーは呼び出し元に返す。握りつぶし禁止
- エラーは `fmt.Errorf("context: %w", err)` でラップしてコンテキストを付ける
- ログは構造化ログ（JSON）を使う。採用ライブラリに統一する
- `interface` は利用側のパッケージで定義する（依存関係逆転）
- テストは `*_test.go`。外部依存はモックする
- `golangci-lint` の警告はすべて解消する

### TypeScript / React

- `any` は原則禁止。型は `types/` に定義する
- API 呼び出しはクライアント層（`lib/api.ts` 等）に集約する。コンポーネント内で直接 fetch しない
- コンポーネントは機能単位（`features/`）と共通 UI（`components/`）を分ける。機能固有のロジックを共通側に混ぜない
- `eslint` / `prettier` の警告はすべて解消する

## ルール（3 段階の境界）

### ✅ Always do

- 変更は小さく：1PR = 1目的。不要な整形はしない
- 変更した領域のテスト・リンタを必ず実行する
- 秘密情報は **絶対にコミットしない**（`.env` / token / secret / webhook URL など）
- ファイルへの言及時は必ずパスを ref として添付する

### ⚠️ Ask first

- 依存追加（npm / Go module 等）や大きなライブラリ導入
- DB スキーマ変更・マイグレーション追加
- 外部サービスの権限・スコープ変更

### 🚫 Never do

- 本番環境への直接操作（デプロイ・設定変更）
- DB 破壊操作（DROP / TRUNCATE）を勝手に実行
- 「テストを消して通す」「ログを消して見えなくする」方向の修正
- 秘密情報・トークンの全文ログ出力

---

## Definition of Done

PR をマージしてよい条件：

1. 変更した領域のテスト・リンタが通る
2. 例外系でユーザに分かるメッセージが返る
3. 秘密情報がコミットに含まれていない
4. 変更ファイル一覧を PR 本文に記載済み
5. `docs/` に運用上の注意が必要なら追記済み
