## 基本

- 日本語で応答すること
- 必要に応じてユーザに質問を行い、要求を明確にすること
- 作業後、作業内容とユーザが次に取れる行動を説明すること
- コマンドの出力が確認できない場合、`get last command` / `check background terminal` を使用して確認すること

## 返信・出力フォーマット

### ファイルへの言及（必ず守る）

Copy permalinkのようなファイルのパスで指定する。
こうすることでgithub上でプレビューが見れて理解が容易になるので

```
# 良い例
`backend/internal/handler/slack.go` の `HandleProgress` 関数を修正しました。

# 悪い例
ハンドラのファイルを修正しました。
```

### 複数ファイルにまたがる変更

変更ファイル一覧を先頭にまとめて提示する：

```
## 変更ファイル
- `path/to/file_a.go` — 変更内容の一言説明
- `path/to/file_b.ts` — 変更内容の一言説明
```

### コードレビューの返し方

レビューコメントは以下の形式で書く：

```
https://github.com/Asheze1127/progress-checker/blob/{コミットハッシュ}/{path/to/file.go}#L{行番号}
問題の説明。（必要なら修正案）
```

- URL は **パーマリンク形式**（ブランチ名ではなくコミットハッシュを使う）で記載する
- コミットハッシュは `git rev-parse HEAD` で取得した値を使う
- 行範囲がある場合は `#L10-L20` の形式で示す

重要度の定義：

| ラベル | 意味 | 対応 |
|--------|------|------|
| `[MUST]` | バグ・セキュリティ・規約違反 | マージ前に対応必須 |
| `[SHOULD]` | 可読性・保守性の改善 | 対応推奨 |
| `[NIT]` | 細かい好みの問題 | 対応任意 |
| `[QUESTION]` | 意図確認 | 回答を求める |

## コーディング規約

### 共通

- コメントは英語で行う
- マジックナンバーはその上にコメントアウトで説明を書く
- TODO / FIXME は `TODO: gh-(対応するissue番号)` の形式で書く
- ファイルを新規作成する前に、既存ファイルで対応できないか確認する
- 不要なコメント（「ここでXXをする」などコードをそのまま説明するだけのもの）は書かない

## ルール（3 段階の境界）

### ✅ Always do

- 変更は小さく：1PR = 1目的。不要な整形はしない
- 変更した領域のテスト・リンタを必ず実行する

### ⚠️ Ask first

- 依存追加（npm / Go module 等）や大きなライブラリ導入
- DB スキーマ変更・マイグレーション追加
- 外部サービスの権限・スコープ変更

### 🚫 Never do

- 本番環境への直接操作（デプロイ・設定変更）
- DB 破壊操作（DROP / TRUNCATE）を勝手に実行
- 「テストを消して通す」「ログを消して見えなくする」などの本質的ではない修正を行わない
- 秘密情報・トークンの全文ログ出力

---

## Definition of Done

PR をマージしてよい条件：

1. 変更した領域のテスト・リンタが通る
2. 例外系でユーザに分かるメッセージが返る
3. 秘密情報がコミットに含まれていない
4. `docs/` に運用上の注意が必要ならドキュメントに追記するようにサジェストする
